A) FIX + UPGRADE SHIFT BLAST (Broadcast Engine)
1) Ensure broadcast works reliably:
- When HR chooses “Broadcast” on a shift (or shift request), the system must:
  - Compute eligible workers (active, role/skill match, not blocked, no conflicting assigned shift, region/radius match if available)
  - Create shift_offers for those workers (pending)
  - Immediately send push notifications to those workers with deep link to the offer screen
- Add server logs + audit_log entries:
  - “SHIFT_BROADCAST_STARTED”
  - “SHIFT_OFFERS_CREATED” {count}
  - “PUSH_SENT” {count}
  - “SHIFT_BROADCAST_FAILED” {error}
- Add a health/debug endpoint for HR admin: /admin/debug/broadcast/:shiftId returning counts of eligible/offers/push attempts.

2) Wave broadcasting (automation):
- Broadcast in waves to reduce noise and increase fill rate:
  - Wave 1: top 20 ranked eligible workers (see Ranking section)
  - If no acceptance after 10 minutes → Wave 2: next 50
  - If still no acceptance after 10 minutes → Wave 3: all remaining eligible
- Implement as scheduled job running every 1–5 minutes:
  - Finds shifts in status “offered” with no acceptance and wave due
  - Sends next wave offers + push
- Must be idempotent: never re-create duplicate offers for same worker+shift.

3) First-accept wins + fill notification:
- When a worker ACCEPTS:
  - Atomically assign shift.assignedWorkerId = workerId, shift.status = assigned
  - Mark that offer accepted
  - Mark all other pending offers as expired/cancelled
  - Notify all other workers with pending offers: “Shift filled by another worker”
  - Update their UI to show “FILLED” (no Accept button)
  - Notify HR + Client: “Worker assigned: {Name}”
- Add concurrency safety: if two accept at same time, only first succeeds; second gets “Already filled”.

B) SMART CLOCK-IN (Clear Reasons + Late Workflow)
1) Worker clock-in eligibility rules (deterministic):
- Define config constants:
  - earlyWindowMinutes (ex: 60)
  - gracePeriodMinutes (ex: 10)
  - lateThresholdMinutes (ex: 10 or 15)
  - allowedRadiusMeters (ex: 250) OR if no GPS feature yet, skip distance check but keep structure.

2) When worker taps “Clock In”:
- If too early: show exact reason + countdown:
  - “Too early. Clock-in opens in 47 minutes.”
- If shift ended: “Shift already ended.”
- If too far (if enabled): “You’re 312m away. Move closer to {Workplace}.”
- If already checked in: “Already clocked in.”
- If late beyond gracePeriod:
  - Show modal requiring reason (dropdown + optional note):
    - Traffic
    - Transportation issue
    - Emergency
    - Other (requires text)
  - Allow clock-in anyway BUT:
    - Mark timesheet entry flaggedLate = true
    - Store lateReason + lateNote
    - Notify HR immediately: “Late clock-in: {Worker} {minutesLate}m. Reason: {reason}”
    - Add audit_log event “LATE_CLOCKIN_REQUESTED”
- If HR approval required (choose this approach):
  - Option A (recommended for now): allow clock-in but flagged for HR review (no blocking)
  - HR can later approve/deny/adjust time in Admin screen.

3) Auto-missed shift detection (automation):
- Scheduled job checks:
  - If shift starts and worker not checked-in by X minutes (ex: 15):
    - Push worker: “Your shift started 15 minutes ago. Are you on your way?”
    - Buttons: “On my way” / “I have an issue”
  - If still no check-in by Y minutes (ex: 30):
    - Notify HR: “Possible no-show: {Worker}”
    - Add audit_log “NO_SHOW_RISK”
- Capture worker responses in shift_checkins table:
  - statuses: on_my_way, issue_reported, checked_in, checked_out
  - timestamps + optional note
- If “I have an issue”:
  - Require short reason + notify HR immediately.
